## 一、开场（30秒）

这个项目来自课程的实验十三——"小车平衡控制综合实验"。目前项目整体已经完成硬件驱动层和软件控制框架的开发，进入了最后的**PID 调参阶段**。

## 二、项目背景（30秒）

**项目目标**：让一个两轮小车像倒立摆一样，仅靠两个轮子就能稳定站立，并且能够抵抗外界干扰。

**核心原理**：通过 MPU6050 陀螺仪+加速度计实时检测车身的倾斜角度和角速度，经过 PID 控制算法计算出电机需要的力矩，驱动电机向倾倒方向加速，把车身"推"回到直立位置。

**验收标准**：
1. 小车能稳定直立，位移不超过 5cm
2. 受到 10° 以内的干扰后能自动恢复平衡
3. 电池电压低于 10V 时自动关闭平衡功能（安全保护）

## 第一章：嵌入式开发基础

### 1.1 什么是嵌入式开发？

嵌入式开发就是**给专用的微型计算机写程序**，让它控制硬件完成特定功能。

和你平时用的电脑不同：
- **电脑**：通用计算，有操作系统（Windows/Linux），可以安装各种软件
- **单片机（MCU）**：专用控制，没有操作系统（或只有极简 RTOS），程序直接操作硬件

一个嵌入式系统通常包括：
```
传感器（感知世界）→ MCU（分析和决策）→ 执行器（做出动作）
     ↑                                    ↑
  MPU6050                            电机 + 轮子
```

### 1.2 STM32 是什么？

STM32 是意法半导体（STMicroelectronics）公司推出的一系列 **32 位 ARM Cortex-M 微控制器**。

本项目使用的 **STM32F103C8T6**：

| 参数 | 含义 |
|------|------|
| F1 | 基础型系列 |
| 03 | 具体型号 |
| C | 48 引脚封装 |
| 8 | 128KB Flash（程序存储器） |
| T | LQFP 封装 |
| 6 | 工业级温度范围 |

**核心参数**：
- **内核**：ARM Cortex-M3，32 位处理器
- **主频**：最高 72MHz（本项目用 64MHz）
- **Flash**：128KB（存程序）+ **RAM**：20KB（存变量）
- **外设**：定时器、ADC、USART、SPI、I2C、GPIO 等

> 打个比方：STM32 就像一个"小电脑"，里面有 CPU（运算）、内存（暂存数据）、硬盘（存程序）、还有各种"接口卡"（外设）用来连接传感器和电机。

### 1.3 开发环境：Keil MDK

**Keil MDK**（Microcontroller Development Kit）是 ARM 公司官方的嵌入式开发 IDE。

**你需要知道的几个关键操作**：

| 操作 | 快捷键 | 说明 |
|------|--------|------|
| 编译 | F7 | 把 C 代码变成机器码 |
| 烧录 | F8 | 把编译好的程序下载到芯片里 |
| 调试 | Ctrl+F5 | 进入在线调试模式，可以打断点 |

**编译产物的含义**：
- `.o` / `.axf`：编译生成的机器码文件
- `.hex`：用于烧录到芯片的十六进制文件
- 编译结果 **0 错误 0 警告** = 代码语法正确，可以烧录

**重要的工程配置**：
- **C99 mode**：使用 C99 语言标准（允许 `//` 注释、`for` 内声明变量等）
- **MicroLIB**：使用 Keil 的精简 C 库（节省代码空间）
- **Optimization -O1**：轻度优化（平衡代码大小和调试便利性）
- **宏定义**：`USE_HAL_DRIVER,STM32F103xB`（告诉编译器使用 HAL 库，芯片型号是 STM32F103xB）

### 1.4 HAL 库是什么？

**HAL** = Hardware Abstraction Layer（硬件抽象层）。

**为什么需要 HAL？**

在不使用 HAL 时，操作硬件需要直接读写寄存器：
```c
// 直接操作寄存器（难读、易错）
GPIOA->BSRR = 0x0010;  // 把 PA4 设成高电平
```

使用 HAL 后，用函数名就能看懂在做什么：
```c
// HAL 库函数（清晰、易读）
HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);  // 把 PA4 设成高电平
```

**HAL 库的三个层级**：
- **HAL 层**：高抽象，功能完备，代码最大
- **LL 层**（Low Layer）：低抽象，接近寄存器，代码小
- **寄存器层**：直接操作寄存器，代码最小但最难读

本项目主要使用 HAL 库，但在代码空间紧张的地方（如 ADC、OLED）改为寄存器级操作以节省 Flash 空间。

### 1.5 本项目开发流程

```
1. 编写 C 代码（main.c + 各驱动模块）
2. Keil MDK 中 F7 编译
3. 通过 ST-Link 仿真器 F8 烧录到芯片
4. 芯片断电重启 → 程序开始运行
5. 串口连接电脑 → 在串口助手中观察 debug 数据
6. 根据观察结果修改 PID 参数 → 重复 2-6
```

## 第二章：STM32 外设详解

### 2.1 GPIO（通用输入输出）

**是什么**：芯片的"引脚"，可以设置为输入模式（读取外部信号）或输出模式（向外发送信号）。

**类比**：GPIO 就像芯片的"手"——想亮灯就把对应的手举起来（输出高电平），想读按键就用手去摸一下（输入检测）。

**本项目中的 GPIO 使用**：

| 引脚 | 模式 | 用途 |
|------|------|------|
| PA4 | 推挽输出 | LED 指示灯 |
| PA5 | 上拉输入 + 中断 | KEY2 按键 |
| PB12~PB15 | 推挽输出 | TB6612 电机方向控制 |
| PA0, PA1 | 浮空输入 | 左编码器 A/B 相 |
| PB6, PB7 | 浮空输入 | 右编码器 A/B 相 |

**GPIO 的关键概念**：

- **推挽输出（Push-Pull）**：既能输出高电平，也能输出低电平，驱动能力强。LED 和电机方向控制用这个模式。
- **上拉输入**：内部接一个电阻到 VCC（3.3V），默认读到高电平，按下按钮拉到 GND 读到低电平。
- **复用推挽输出（AF_PP）**：GPIO 不归 CPU 直接控制，而是交给外设（如定时器的 PWM 输出）。PA8 的 PWM 就用这个模式。

### 2.2 定时器（Timer）

**是什么**：一个硬件计数器，每个时钟周期自动 +1，计到设定值后可以从头再来。可以产生精准的时间间隔。

**本项目中的四种定时器用法**：

#### 用法一：PWM 输出（TIM1）

**PWM** = Pulse Width Modulation（脉宽调制）

**原理**：用方波的"占空比"来控制电机的平均电压。

```
周期 = 7200 个计数（对应 10KHz）

占空比 0%：  ____ ____ ____    → 电机不转
占空比 50%： --__--__--__--__  → 电机半速
占空比 100%：------______------ → 电机全速
```

**关键参数**：
- `Prescaler = 0`：不分频，72MHz 直接给定时器
- `Period = 7199`：计满 7200 个时钟 = 一个 PWM 周期
- 所以 PWM 频率 = 72MHz / 7200 = **10KHz**

> 为什么选 10KHz？太低（<1KHz）电机会发出刺耳啸叫，太高开关损耗大。10KHz 是人耳听不到的频率。

**本项目用 TIM1 的 CH1（PA8）和 CH4（PA11）分别控制左右电机的 PWM。**

#### 用法二：编码器模式（TIM2、TIM4）

编码器模式是 STM32 定时器的一个特殊功能。它可以直接读取正交编码器的 A/B 两相脉冲，自动识别方向和计数。

**正交编码原理**：
```
A 相：__--__--__--__--
B 相：--__--__--__--__  （B 相比 A 相滞后 90°，即"正交"）

正转：A 先于 B → 计数器自动 +
反转：B 先于 A → 计数器自动 -
```

你不用手动判断方向，定时器硬件会自动处理。代码只需调用：
```c
int val = (int16_t)__HAL_TIM_GET_COUNTER(htim);  // 读计数值
__HAL_TIM_SET_COUNTER(htim, 0);                   // 清零
```

这样每 5ms 读到的 `val` 就代表这 5ms 内电机转了多少（带方向）。

#### 用法三：周期中断（TIM3）

TIM3 用于产生 **5ms 定时中断**：
- `Period = 50 - 1 = 49`：计 50 个数触发一次中断
- 中断频率 = 10KHz / 50 = **200Hz**
- 在中断中读取编码器值、计算速度

#### 用法四：SysTick（系统滴答定时器）

Cortex-M3 内核内置的 24 位定时器，用于产生精准延时：
```c
delay_init(SystemCoreClock / 1000000);  // 初始化，1μs 精度
delay_ms(500);   // 延时 500 毫秒
delay_us(100);   // 延时 100 微秒
```

### 2.3 中断（Interrupt）

**是什么**：CPU 正在执行主程序时，硬件突然说"有急事！"，CPU 暂停手头工作去处理紧急事件，处理完再回来继续。

**类比**：你在看书（主程序），手机响了（中断发生），你放下书接电话（中断处理），挂电话后继续看书。

**本项目中的三个中断**：

| 中断源 | 引脚 | 优先级 | 功能 |
|--------|------|--------|------|
| EXTI15_10 | PA12（MPU6050 INT） | 2 | **200Hz 控制核心**：读姿态 → 互补滤波 → PID 计算 → 驱动电机 |
| EXTI9_5 | PA5（KEY2） | 2 | 按键启停电机，30ms 消抖 |
| TIM3 | 内部定时器 | 1 | 每 5ms 读取编码器、计算速度 |

**NVIC（嵌套向量中断控制器）**：管理所有中断的"调度员"。
- 每个中断有优先级（数字越小优先级越高）
- 高优先级中断可以打断低优先级中断
- 本项目 TIM3（优先级 1）比 EXTI（优先级 2）更高

**中断的完整流程**：
```
硬件触发 → NVIC 判断优先级 → 保存现场 → 跳转到 IRQHandler
  → 调用 HAL 回调 → 执行你的代码 → 恢复现场 → 继续主程序
```

### 2.4 通信协议

#### I2C（MPU6050）

**是什么**：一种两线制的串行通信协议，只需要两根线。
- **SCL**（时钟线）：主设备控制节奏
- **SDA**（数据线）：双方在 SCL 的节拍下传输数据

**为什么用软件 I2C？**

硬件 I2C 由芯片内置的外设自动处理，但 STM32F1 的硬件 I2C 有历史遗留的 bug。所以本项目用"软件模拟 I2C"：手动拉高拉低 GPIO 来模拟时序。代码在 `atk_ms6050_iic.c` 中。

**MPU6050 的数据如何读取？**
```
1. STM32 通过 I2C 发送"我要读加速度计 X 轴"的指令
2. MPU6050 返回 2 个字节的原始数据（高字节 + 低字节）
3. 把原始数据 ÷ 量程 = 实际的 g 值
```

#### SPI（OLED）

**是什么**：四线制同步串行通信，比 I2C 快。
- **SCK**（时钟）
- **MOSI**（主发从收，Master Out Slave In）
- **CS**（片选，选中哪个设备）
- **DC**（数据/命令选择）

本项目用 4 线 SPI 连接 OLED 显示屏（SSD1306 控制器）。

#### UART（串口调试）

**是什么**：异步串行通信，电脑上最常用的调试接口。

**配置**：115200 波特率, 8 数据位, 1 停止位, 无校验（简称 115200 8N1）

**printf 重定向**：标准 C 库的 `printf()` 默认输出到屏幕，在嵌入式里没有屏幕。我们把 `printf` 重定向到 USART1，这样 `printf("hello")` 就会通过 PA9 引脚发送到电脑的串口助手上显示。

**本项目串口输出格式**（兼容 MicroLIB，全部使用整数）：
```
D,时间,pitch×100,gyro×10,左编码器,右编码器,速度和,直立环PWM,速度阻尼PWM,左目标,右目标,左实际,右实际,电机开关,平衡使能
```

## 第三章：传感器与执行器

### 3.1 MPU6050 — 六轴姿态传感器

**是什么**：集成了三轴加速度计 + 三轴陀螺仪的 MEMS 传感器。

#### 加速度计

**原理**：测量每个轴上的"加速度"（包括重力加速度）。静止时，重力加速度的方向就是"向下"的方向，据此可以算出倾角。

```
       /|
      / |  车身倾斜 θ 度
     /  |
    /   | 重力 g
   /    |
  /_____|

加速度计测倾角：θ = atan2(ay, az)
```

**问题**：加速度计对振动非常敏感！电机一转，加速度计读到的就不只是重力了，还有电机振动带来的加速度。所以单独用加速度计算角度会非常抖。

#### 陀螺仪

**原理**：测量绕每个轴的"旋转角速度"（单位：°/s）。

对陀螺仪角速度做积分，可以获得角度：
```
角度 = 累加(角速度 × 时间间隔)
```

**问题**：积分会累积误差。陀螺仪有"零偏"（静止时也有微小的输出），积分时间长了误差会越来越大。

#### 互补滤波 — 融合两者优点

**核心思想**：加速度计低频准但高频噪声大；陀螺仪高频准但低频漂移。互补滤波让：
- 在**短时间内**（高频），相信陀螺仪（积分）
- 在**长时间内**（低频），相信加速度计（修正漂移）

**公式**：
```c
angle = 0.95 × (angle + gyro × dt) + 0.05 × accel_angle
//       ↑                          ↑
//       95% 信任陀螺仪积分          5% 信任加速度计修正
```

**α=0.95 的含义**：
- 每次计算，95% 来自陀螺仪积分（上次角度 + 角速度×时间）
- 5% 来自加速度计的反正切结果（修正长期漂移）
- 时间常数 τ ≈ dt / (1-α) ≈ 5ms / 0.05 ≈ **95ms**

**α 的调节**：
- α 越大（如 0.98）：更平滑，但响应慢，滞后大
- α 越小（如 0.90）：响应快，但噪声多

#### 本项目 MPU6050 的关键配置

| 配置项 | 值 | 含义 |
|--------|-----|------|
| 陀螺仪量程 | ±2000°/s | 满量程，适合快速运动 |
| 加速度计量程 | ±2g | 高精度，适合测倾角 |
| 采样率 | 200Hz | 每 5ms 输出一个中断 |
| DLPF | 默认 98Hz | 低通滤波带宽 |
| 轴组合 | AY + GX | **由 MPU6050 物理安装方向决定，不可随意修改** |

### 3.2 正交编码器 — 电机测速

**是什么**：装在电机轴上的传感器，电机转一圈输出固定数量的 A/B 两相脉冲。

**工作原理**：
- 电机轴上有一个带光栅的码盘
- 两个光电传感器（A 和 B）错开 90° 放置
- 电机转动时，A 和 B 交替输出脉冲
- A 先 B 后 = 正转，B 先 A 后 = 反转

**STM32 编码器模式**：
- 配置定时器为"编码器模式"
- A 相接 CH1，B 相接 CH2
- 定时器的计数器会自动根据 A/B 的相位关系 +1 或 -1
- 不需要软件判断方向，硬件全自动！

**本项目速度计算**（每 5ms 执行一次）：
```c
encoder_left = Read_Encoder(&htim2);   // 读取并清零左编码器计数
encoder_right = Read_Encoder(&htim4_encoder);  // 读取并清零右编码器计数
```

`encoder_left` 的值 = 这 5ms 内左电机转了多少"编码器刻度"。

### 3.3 TB6612 — 电机驱动芯片

**是什么**：双路 H 桥电机驱动芯片，可以控制两个直流电机的转速和方向。

**为什么需要驱动芯片？**
STM32 的 GPIO 只能输出 3.3V、几十 mA 的信号，不足以驱动 12V、几百 mA 的电机。TB6612 相当于一个"功率放大器"。

**TB6612 的控制逻辑**（以左电机为例）：

| AIN1 (PB14) | AIN2 (PB15) | PWM (PA8) | 结果 |
|-------------|-------------|-----------|------|
| 1 | 0 | X | 正转，转速 = X/7200 × 100% |
| 0 | 1 | X | 反转，转速 = X/7200 × 100% |
| 1 | 1 | — | 刹车（电机两端短路制动） |
| 0 | 0 | — | 滑行（电机断开，自由转动） |

### 3.4 ADC — 电池电压检测

**是什么**：模拟-数字转换器（Analog to Digital Converter）。把连续的模拟电压转换成离散的数字值。

**为什么需要？**
电池电压是模拟量（比如 11.7V），芯片只能处理数字量（0 和 1）。ADC 把电压变成数字。

**电阻分压**：
```
电池 +12V
  │
  ├── R1（上电阻，约 10KΩ）
  │
  ├── PB0（ADC 输入引脚）→ 读到的电压
  │
  ├── R2（下电阻，约 33KΩ）
  │
 GND

分压公式：V_PB0 = V_battery × R2 / (R1 + R2)
例如：12V × 33/43 ≈ 9.2V → 但 ADC 只能读 0~3.3V！

所以实际上 R1 和 R2 的比例是：
电池 12.6V → 分压后 3.3V → ADC 读满量程 4095
```

**STM32F103 ADC 参数**：
- 12 位分辨率 → 4096 个级别（0~4095）
- 参考电压 3.3V → 每个级别 = 3.3V / 4096 ≈ 0.8mV
- ADC_IN8 通道（PB0 引脚）

**本项目 ADC 的特殊处理**：
- 为了节省 Flash 空间，ADC 改用了**寄存器级驱动**（不是 HAL 库）
- 直接操作 `ADC1->CR1`, `ADC1->CR2`, `ADC1->SQR` 等寄存器

### 3.5 OLED 显示屏

**是什么**：0.96 英寸 128×64 像素的 OLED 屏幕，控制器是 SSD1306。

**显示内容（4 行）**：
```
Pitch: -0.25     ← 车身倾角
Speed: 15        ← 左右编码器速度之和
Batt: 12.3V      ← 电池电压
Motor: ON        ← 电机启停状态
```

**关键实现细节**：
- OLED 的几个控制引脚（PB3, PB4, PA15）与 JTAG 调试口冲突
- 代码中调用 `__HAL_AFIO_REMAP_SWJ_NOJTAG()` 释放这些引脚给 OLED 使用
- **烧录后不能直接用 JTAG 调试**（因为引脚被 OLED 占用了），但 SWD 模式仍然可用
- OLED 驱动也是**寄存器级**的（用 BSRR/BRR 寄存器直接操作 GPIO）
- **必须调用 `oled_refresh_gram()` 才能刷新显示**，否则画面不更新

## 第四章：PID 控制算法 — 项目的核心

### 4.1 什么是 PID？

PID 是最经典、最广泛使用的自动控制算法。它根据"当前误差"、"过去误差"和"未来趋势"三个维度来计算控制输出。

```
         ┌─────────┐
目标 ──→│ PID 控制 │──→ 执行器 ──→ 被控对象 ──→ 实际输出
   ↑     └─────────┘                           │
   └───────────────────────────────────────────┘
                  反馈（传感器）
```

### 4.2 三个分量详解

#### P（比例 — Proportional）

**作用**：根据**当前**误差大小，按比例输出控制量。

```c
balance_pwm = Kp × pitch;  // 倾角越大，PWM 越大
```

**直观理解**：像一根弹簧。倾角偏离 0° 越远，回推的力越大。

**Kp 太大**：反应过度，来回剧烈震荡 → 高频抖动
**Kp 太小**：力道不够，站不住 → 一松手就倒

#### D（微分 — Derivative）

**作用**：根据误差的**变化速度**（角速度），提前抑制。

```c
balance_pwm = Kp × pitch + Kd × gyro_y;
//                         ↑
//                角速度 gyro_y 就是 pitch 的变化率
```

**直观理解**：像阻尼器（减震器）。当车正在快速倒下时，角速度很大，D 项产生一个反向力"刹车"，阻止倒下的趋势。

**Kd 太大**：反应迟钝，推都推不动；电机高频吱吱响（放大了陀螺仪噪声）
**Kd 太小**：车来回摇晃，像不倒翁 → 低频晃动

#### I（积分 — Integral）

**作用**：累积**过去所有的误差**，消除稳态偏差。

```
积分 = 累加(每次的误差)
control += Ki × 积分
```

**直观理解**：如果你站了很久身体都微微前倾，P 项已经不足以纠正这个微小偏移，I 项会慢慢累积，最终"推一把"让你回到正中间。

**Ki 太大**：积分饱和 → 小车突然猛冲
**Ki 太小**：消除静差太慢

**本项目当前状态**：I 项暂时关闭（= 0），因为：
1. 先要把直立环 PD 调稳定
2. 速度环的 I 很容易和直立环"打架"，让平衡点偏移

### 4.3 串级 PID — 本项目控制架构

```
外环（速度环）— 慢，关心"小车跑远了没有"
  │
  │ 速度误差 → Velocity_Kp × 误差 → 输出叠加
  │
  ▼
内环（直立环）— 快，关心"车身歪了多少"
  │
  │ 角度误差 → Balance_Kp × 角度 + Balance_Kd × 角速度 → 输出
  │
  ▼
电机 PWM = 内环输出 + 外环输出
```

**为什么串级？**
- 直立环（内环）频率 200Hz，响应极快，优先保证"不倒下"
- 速度环（外环）叠加在直立环上，体现在的效果是"车在直立的基础上慢慢回位"
- 如果先把速度环开得太大，车为了"停在原地"会牺牲"直立"，反而会倒

**这就是"先内环后外环"的根本原因。**

### 4.4 增量式 PI（速度环）

本项目速度环使用**增量式 PI**，不同于普通的"位置式 PI"：

**位置式**（直接用累积误差）：
```c
output = Kp × error + Ki × integral;
integral += error;
```

**增量式**（只用误差的变化）：
```c
delta = Kp × (error - last_error) + Ki × error;  // 计算增量
output += delta;  // 在上次基础上加增量
```

增量式的优点：
- 积分自然有界（不会无限制增长）
- 电机启停时切换更平滑
- 更容易实现抗积分饱和

### 4.5 调参方法论

**总原则：一次只改一个参数！**

```
第一步：调 Balance_Kp
  Kd=0, 从 100 开始，每次加 20~50
  找到"松手能短暂站住但有点晃"的值

第二步：调 Balance_Kd
  Kp 固定, Kd 从 0.5 开始，每次加 0.2~0.5
  找到"不抖也不晃"的值

第三步：反复微调 Kp/Kd
  在最佳值附近 ±20% 微调

第四步：加速度环 Vel_Kp
  从 15 开始，缓慢增加

第五步：加 Vel_Ki（最后最后）
  Ki = Kp / 200，宁小勿大
```

**现象诊断速查表**：

| 现象 | 原因 | 怎么办 |
|------|------|--------|
| 电机高频滋滋响 | Kd 太大 | 减小 Kd |
| 车左右缓慢大幅摇摆 | Kd 太小 | 增大 Kd |
| 松手立刻倒下，电机无力 | Kp 太小 | 增大 Kp |
| 松手后剧烈来回抖动 | Kp 太大 | 减小 Kp |
| 站得住但一直往一个方向跑 | 机械零点不准 | 修正 MECHANICAL_ZERO |
| 站得住但越跑越快 | 缺少速度环 | 增大速度阻尼或加 Vel_Kp |
| PWM 一直顶到最大还不回正 | 限幅太小 或 Kp 不够 | 调大限幅或 Kp |
| 按 KEY2 没反应 | 电机没开启 | OLED 确认 Motor: ON |

## 第五章：项目工程要点

### 5.1 时钟系统

STM32F103 的时钟树是理解系统的基础：

```
HSI（内部 8MHz 振荡器）         HSE（外部 8MHz 晶振）
       │                              │
       ├── HSI/2 = 4MHz               ├── HSE = 8MHz
       │                              │
       └── PLL ×16 = 64MHz  ←──      └── PLL ×9 = 72MHz
              │                              │
              └── SYSCLK（系统时钟）──────────┘
                     │
                     ├── HCLK = SYSCLK（CPU 和总线）
                     ├── PCLK1 = HCLK/2（APB1，TIM2/3/4）
                     └── PCLK2 = HCLK（APB2，TIM1/GPIO）
```

**本项目为什么用 HSI+PLL=64MHz 而不是 HSE+PLL=72MHz？**

因为在实际调试中发现：
- 板上的外部 HSE 晶振可能不存在、损坏或频率不是标准的 8MHz
- 如果 HSE 频率不准，USART 波特率就会跑偏 → 串口乱码
- 所以改为强制使用内部 HSI，虽然慢了一点（64MHz vs 72MHz），但**串口稳定可靠**

### 5.2 代码大小优化

STM32F103C8T6 只有 128KB Flash，而 MDK 评估版限制 32KB。为了挤进这个限制，做了大量优化：

| 优化 | 方法 | 节省 |
|------|------|------|
| MicroLIB | 使用 Keil 精简 C 库 | ~5KB |
| ADC 寄存器化 | 不链接 HAL_ADC.c，直接写寄存器 | ~2.7KB |
| DMA 空桩 | 不链接 HAL_DMA.c，用空函数替代 | ~1.7KB |
| 字体精简 | OLED 只保留 16×16 字体，删除 12×12 和 24×24 | ~450B |
| -O1 优化 | 比 -O0 减小代码 | ~30% |
| 移除未用 HAL | 不链接 tim_ex, adc_ex, rcc_ex, pwr, spi 等 | ~2KB |

### 5.3 电机死区

**什么是死区？**

直流电机有一个特性：PWM 太小时电机不转（摩擦力 > 驱动力）。这个"不转的最小 PWM"就是死区。

例如你的左电机需要 PWM ≥ 285 才开始转，那 0~284 就是"死区"。

**为什么要补偿死区？**

在平衡控制中，PID 经常输出很小的 PWM（比如 ±50~±200）。如果不补偿死区，小倾角时电机完全不动 → 等角度大了才猛冲 → 过度纠正 → 又反向猛冲 → 震荡。

**本项目的死区处理方法**：

经过多次实验（包括专门的 test2 死区测试程序），采用了"柔和死区"方案：

```c
if (pwm > 5)       pwm += 100;   // 正向需要电机转，叠加上死区偏移
else if (pwm < -5)  pwm -= 100;  // 反向需要电机转，叠加上死区偏移
else                pwm = 0;     // 信号太小，不转
```

- START=5：只有超出 ±5 的信号才驱动电机（避免噪声误触发）
- DZ=100：一旦决定要转，额外加 100 的 PWM 来克服静摩擦
- 这个值是根据 test2 实测得到的

### 5.4 机械零点

**是什么**：小车"物理上直立"时，MPU6050 读到的角度值。

理想情况下，车完美直立时 pitch = 0.00°。但实际上：
- MPU6050 焊接时不可能完全水平
- 电池、电路板的重心偏移
- 导致"物理直立"≠"传感器 0°"

**如何校准**：
1. 用手扶住小车，找到它能站稳的倾角
2. 观察 OLED 上显示的 pitch 值
3. 把这个值填入 `MECHANICAL_ZERO`

当前使用的值：**-0.10**（经过多次实测调整）

### 5.5 安全保护机制

| 保护 | 实现 | 说明 |
|------|------|------|
| 角度过大关停 | `if (pitch > 30° or pitch < -30°)` 关闭电机 | 防止倒地后轮子疯转 |
| PWM 限幅 | `PWM_Limit()` 限制在 ±7199 | 防止溢出 |
| 电机安全限幅 | `MOTOR_SAFE_LIMIT` | 独立于 PWM 限幅的额外保护 |
| 低压保护 | 暂时绕过（需校准 ADC 后恢复） | 电池 < 10V 时关闭平衡 |
| 抗积分饱和 | Vel_PI 函数内限幅到 `[PWM_MIN, PWM_MAX]` | 防止积分项无限增长 |
| 启停保护 | 电机关闭时清零积分项 | 防止重新开启时猛冲 |

## 第六章：调试与数据观察

### 6.1 串口 CSV 调试数据

串口输出格式（每 50ms 一行）：
```
D,t,p100,g10,sl,sr,ss,bp,sp,fl,fr,ml,mr,en,be
```

**实际数据示例**：
```
D,12345,75,-25,0,12,12,-195,0,195,195,295,295,1,1
```

解读：系统运行 12.345 秒，车身前倾 0.75°，角速度 -2.5°/s（正在往后转），左轮没转右轮转了 12 格，直立环输出 -195，速度环输出 0（关闭），左右目标 PWM 各 195，经死区补偿后实际输出 295，电机开启，平衡使能。

**如何使用这些数据调参？**

| 观察 | 含义 |
|------|------|
| `p100` 在 ±200 来回跳 | 震荡，Kp 太大或 Kd 太小 |
| `fl/fr` 和 `ml/mr` 差很多 | 死区在起作用 |
| `ml/mr` 长时间顶到 ±800 | 限幅卡住，需提高限制或 Kp |
| `sl` 一直为 0 | 左编码器有硬件问题 |
| `g10` 剧烈波动 | 陀螺仪噪声大，考虑调 DLPF |

### 6.2 OLED 实时显示

4 行关键数据的实时显示让你不需要接电脑也能看到小车状态：
- **Pitch**：第一行，最直观——能看到车是否在震荡
- **Speed**：编码器速度之和，为 0 最好
- **Batt**：电池电压
- **Motor**：ON/OFF，确认电机状态
