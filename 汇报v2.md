# 第二部分：文件分类详解

## 一、纯库文件（不需要修改，只需"使用"）

这些文件来自 ST 官方固件包（STM32Cube_FW_F1_V1.8.0），**一行都不需要改**，只做编译链接。

### 1. CMSIS 层 — CPU 内核标准接口

```
Drivers/CMSIS/Include/core_cm3.h         ← Cortex-M3 内核定义（寄存器地址、指令）
Drivers/CMSIS/Include/cmsis_*.h          ← ARM 统一的 CMSIS 标准
Drivers/CMSIS/Device/ST/STM32F1xx/
  ├── stm32f103xb.h                      ← 芯片寄存器地址映射（GPIOA 地址、TIM1 地址等）
  ├── stm32f1xx.h                        ← 按编译宏选择对应型号的头文件
  └── system_stm32f1xx.c                 ← 系统启动初期配置（默认时钟）
```

**这些文件做什么**：把芯片的物理地址（如 `0x40010800` = GPIOA 基地址）翻译成 C 语言可以用的宏。例如你写 `GPIOA->BSRR`，就是 CMSIS 帮你定义好的。

### 2. HAL 库 — 外设驱动封装

```
Drivers/STM32F1xx_HAL_Driver/
  ├── stm32f1xx_hal.c/h           ← HAL 核心（初始化、Tick）
  ├── stm32f1xx_hal_gpio.c/h      ← GPIO 驱动
  ├── stm32f1xx_hal_rcc.c/h       ← 时钟树配置
  ├── stm32f1xx_hal_tim.c/h       ← 定时器驱动（PWM、编码器都靠它）
  ├── stm32f1xx_hal_uart.c/h      ← 串口驱动
  ├── stm32f1xx_hal_exti.c/h      ← 外部中断驱动
  ├── stm32f1xx_hal_cortex.c/h    ← NVIC 中断管理
  ├── stm32f1xx_hal_flash.c/h     ← Flash 操作
  └── stm32f1xx_hal_dma.c/h       ← DMA 驱动（本项目只用空桩）
```

**这些文件做什么**：把操作寄存器的繁琐过程封装成函数。比如要配置一个 GPIO 为输出，直接操作寄存器需要理解 4-5 个寄存器位域，而 HAL 只需填一个结构体调用 `HAL_GPIO_Init()`。

**我做了什么**：只改了一个配置头文件 `stm32f1xx_hal_conf.h`（开关哪些 HAL 模块参与编译），**不改 HAL 源代码**。

### 3. 启动文件

```
Startup/startup_stm32f103xb.s   ← 汇编写的启动代码
```

**做什么**：芯片上电后最先执行的代码。初始化堆栈、中断向量表、调用 `main()`。**完全不需要改。**

# 第三部分：自写代码的设计思路与技术剖析

## 总体设计思路：分层架构

我按照**分层架构**的思想组织代码，从上到下依次是：

```
应用层（main.c）
  │  负责：初始化顺序、主循环（OLED显示 + 串口输出）
  │  依赖：下面所有层
  ▼
控制算法层（control.c + gtim.c）
  │  负责：PID 计算、速度闭环
  │  依赖：传感器数据（pitch, gyro_y）+ 执行器接口（motorMove）
  ▼
板级驱动层（LED/KEY/EXTI/MOTOR/ENCODER/MPU6050/ADC/OLED）
  │  负责：把硬件操作封装成有意义的函数
  │  依赖：HAL 库 + CMSIS
  ▼
HAL 库 + CMSIS
  │  负责：操作寄存器、提供外设 API
  ▼
硬件（STM32 芯片 + 外设）
```

**为什么要分层？**
- 修改 PID 参数只改 `control.c`，不用碰驱动
- 换一个型号的电机驱动芯片只改 `motor.c`，不用碰控制逻辑
- 每一层可以独立调试

## 逐文件设计详解

### 1. LED 驱动（led.c）— 最简单的 BSP 层示例

**技术点**：GPIO 推挽输出

**设计思路**：把"操作一个引脚"封装成 `led_on()`, `led_off()`, `led_toggle()` 三个函数。上层代码不需要知道 LED 接在 PA4，只需要调用 `led_on()`。

```c
// 初始化：配置 PA4 为推挽输出
// 控制：写 BSRR 寄存器 = 设置引脚电平
// 关键点：低电平点亮（LED 负极接 GPIO，正极接 VCC）
```

**为什么不直接用 `HAL_GPIO_WritePin`？**
可以，但封装后代码更清晰。`led_on()` 比 `HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET)` 更好读。

### 2. 按键驱动（key.c）— 软件消抖

**技术点**：GPIO 输入 + 状态机消抖

**设计思路**：机械按键按下时会抖动（几 ms 内反复通断），用软件消抖——第一次检测到按下后等 10ms 再确认一次。

```
状态机：
  等待按下 → 检测到低电平 → 等 10ms → 再检测
    ↑                    ↓ 仍然低 = 确认按下
    └────────────────────┘ 变回高 = 抖动，忽略
```

**用到的技术**：`static` 变量保持跨调用状态（`key_up` 标志位），简单的两状态状态机。

### 3. 外部中断 + 姿态解算（exti.c）— ★ 系统的"心脏"

**技术点**：EXTI 外部中断、互补滤波、软件消抖、系统调度

这是整个项目**最重要的文件**之一，它是控制循环的入口。每 5ms 执行一次。

**设计思路**：把"传感器读取 → 数据融合 → 控制计算"这条链路的入口放在中断中，保证严格的 5ms 周期。

**核心代码拆解**：

```
HAL_GPIO_EXTI_Callback(GPIO_PIN_12)
  │  触发源：MPU6050 INT 引脚拉低（每 5ms 一次）
  │
  ├─ ① 读传感器 + 姿态解算
  │     read accelerometer → ay, az
  │     read gyroscope     → gx
  │     accel_angle = atan2f(ay, az) × 57.3    ← 加速度计倾角（反正切）
  │     gyro_y = gx / 16.4                      ← 陀螺仪角速度（量程换算）
  │     comp_angle = 0.95×(prev + gyro×dt) + 0.05×accel  ← 互补滤波
  │     pitch = comp_angle
  │
  ├─ ② 电池电压检测（每 100 次 = 500ms 更新一次）
  │     volt_cnt++ → 达 100 → readVolt(10) → battery_voltage
  │
  └─ ③ 调用控制算法
        Balance_Control()  ← 在 control.c 中实现
```

**关键技术：互补滤波**

| 作用 | 为什么需要 |
|------|-----------|
| 加速度计算倾角 | 静态精确，但电机振动会让它剧烈抖动 |
| 陀螺仪积分算角度 | 动态平滑，但零点漂移会让角度慢慢偏移 |
| **互补滤波融合两者** | 95%信任陀螺仪（短时准）+ 5%信任加速度计（长时准） |

公式：`angle = α×(angle + gyro×dt) + (1-α)×accel`

**关键技术：轴组合选择**

MPU6050 的物理安装方向决定了用哪个轴计算倾角。本项目用的是 **AY + GX**（加速度计 Y 轴 + 陀螺仪 X 轴），这是在 Session 3 中经过实测确定下来的（最初误用了 AX+GY）。

**关键技术：按键消抖（PA5）**

在同一个 `HAL_GPIO_EXTI_Callback` 中处理 PA5 中断，用时间窗口消抖（30ms 内重复触发则忽略）。与 key.c 中的 10ms 延时消抖不同，这里是中断上下文不能延时，所以用时间戳比较。

### 4. 电机驱动（motor.c）— ★ 死区补偿是核心难点

**技术点**：PWM 输出、H 桥控制、死区补偿、增益校准

**设计思路**：接收控制算法的 PWM 目标值，经过限幅 → 增益校准 → 死区补偿 → 写入硬件。

**核心流程**：
```
motorMove(speedL, speedR)
  │
  ├─ ① PWM_Limit()         — 限幅到 ±7199
  ├─ ② 增益校准            — × MOTOR_LEFT_GAIN / MOTOR_RIGHT_GAIN
  ├─ ③ motor_deadzone()    — 死区补偿（最关键）
  ├─ ④ MOTOR_SAFE_LIMIT    — 二次限幅
  └─ ⑤ 设置方向 + 写 PWM   — AIN1/AIN2 控制方向，TIM1->CCR 控制占空比
```

**关键技术：PWM + 方向控制**

```
正转：AIN1=1, AIN2=0, PWM=duty
反转：AIN1=0, AIN2=1, PWM=duty
刹车：AIN1=1, AIN2=1 （电机两端短路制动）
滑行：AIN1=0, AIN2=0 （电机断开，自由转动）
```

**关键技术：死区补偿（项目中最难调的部分之一）**

**问题**：直流电机有静摩擦力，PWM < 某个阈值时电机根本不转。实测左电机死区 ~285，右电机 ~280。如果不补偿，小倾角时 PID 输出小 PWM → 电机不动 → 角度继续增大 → PID 突然输出大 PWM → 电机猛冲 → 过度纠正 → 震荡。

**我尝试过的三种方案**：

| 方案 | 逻辑 | 结果 |
|------|------|------|
| 硬阈值 | `if (0 < pwm < 285) pwm = 285` | 小角度完全无响应，1° 以内的倾角被忽略 |
| 偏移式 | `if (pwm > 0) pwm += 285` | 小角度响应好，但高频微抖时电机来回猛冲，"手按不住" |
| **柔和死区（当前）** | `pwm > 5 → pwm += 100`，`pwm < 5 → pwm = 0` | 妥协方案：小信号不反应（靠自然摩擦），中等以上信号加大力度跳过死区 |

**设计思想**：死区补偿没有"完美方案"，需要在"小角度响应"和"不产生震荡"之间做权衡。当前方案是经过 **test2 死区测试程序**（单独扫描 PWM 280-320 找到电机恰好转动的阈值）和实际平衡测试后确定的。

**关键技术：增益校准**

左右电机由于制造差异，相同 PWM 下转速不同。用 `MOTOR_LEFT_GAIN` 和 `MOTOR_RIGHT_GAIN` 补偿。

### 5. 编码器驱动（encoder.c）— 正交编码测速

**技术点**：定时器编码器模式

**设计思路**：利用 STM32 定时器硬件的编码器模式，让硬件自动处理 A/B 相信号的识别和计数。

```
初始化：
  ├─ GPIO: PA0/PA1 (左), PB6/PB7 (右) → 浮空输入
  ├─ TIM2: 编码器模式, Period=65535 → 左编码器
  └─ TIM4: 编码器模式, Period=65535 → 右编码器

每 5ms 读取：
  ├─ (int16_t)__HAL_TIM_GET_COUNTER(htim) → 读取并符号化
  └─ __HAL_TIM_SET_COUNTER(htim, 0)       → 清零
```

**关键技术：为什么用 `int16_t` 强转？**

定时器计数器是 16 位的（0~65535），但编码器可能反转（负值）。用 `int16_t` 强制转换，相当于把 65535 解释为 -1、65534 解释为 -2……这样 5ms 内的反转也能正确表示。

**关键技术：左右编码器使用了不同定时器**

这是一个踩过的坑。最初左右编码器用了 TIM2 和 TIM3，但 TIM3 的一个通道（PA7）和 OLED 的 CS 引脚冲突了。于是把右编码器改为 TIM4（PB6/PB7），把 TIM3 腾出来做 5ms 控制定时器。

### 6. ADC 驱动（adc.c）— 寄存器级实现

**技术点**：ADC 寄存器操作、电阻分压、软件校准

**设计思路**：电池电压检测只需要最简单的 ADC 功能（单通道、单次转换、软件触发），HAL_ADC 为这种简单场景引入了 2.7KB 代码，不划算。于是**直接用寄存器操作 ADC 外设**。

**为什么用寄存器而不是 HAL？**

STM32F103C8T6 的 Flash 只有 128KB，MDK 评估版限制 32KB。HAL_ADC 占 2.7KB，在总量只有 636 字节剩余空间的情况下必须砍掉。

**寄存器操作流程**：
```
1. RCC->APB2ENR  |= ADC1EN       — 开启 ADC1 时钟
2. GPIOB->CRL    &= ~(CNF0|MODE0) — PB0 设为模拟输入
3. ADC1->CR1     = 0              — 无扫描、无中断
4. ADC1->CR2     = ADON|EXTSEL|EXTTRIG — 启用、软件触发
5. ADC1->SQR3    = 8              — 通道 8（PB0 = ADC1_IN8）
6. ADC1->SMPR2   = 28.5周期       — 采样时间
7. 校准（RSTCAL → CAL）
```

**电压换算**：
```c
volt = adc_val × 3.3V / 4096 × (R1+R2)/R2
     = adc_val × 3.3 / 4096 × 43/10
// 43 = 总电阻（10K上 + 33K下），10 = 下电阻（33K？实际需万用表校准）
```

### 7. 定时器 + 速度环（gtim.c）— ★ 增量式 PI

**技术点**：定时器周期中断、增量式 PI 控制器

**设计思路**：用 TIM3 产生 5ms 中断，在中断中读取编码器值并保存到全局变量。同时提供 `Vel_PI()` 函数实现速度闭环。

**TIM3 配置**：
```
Prescaler = SystemCoreClock/10000 - 1    ← 自适应时钟频率
Period    = 49                            ← 10000Hz/50 = 200Hz
→ 每 5ms 触发一次 HAL_TIM_PeriodElapsedCallback
  → encoder_read_all()                   ← 读取编码器
```

**关键技术：增量式 PI 控制器**

```
传统（位置式）PI：
  integral += error               ← 误差不断累积
  output = Kp×error + Ki×integral ← 可能非常大

增量式 PI（本项目使用）：
  delta = Kp×(error - last_error) + Ki×error  ← 只算变化量
  output += delta                              ← 在上次输出基础上叠加
```

**为什么用增量式？**
- 积分自然不会无限增长（只有变化量被加进去）
- 电机开关切换时更平滑（没有累积的积分"债"要还）
- 抗积分饱和更简单（直接限幅 output 即可）

**关键技术：抗积分饱和**

```c
if (pwm_out > PWM_MAX) pwm_out = PWM_MAX;
if (pwm_out < PWM_MIN) pwm_out = PWM_MIN;
```

输出被限幅后，下次计算 `delta` 时只在限幅后的值上叠加，防止积分项无限膨胀。

**关键技术：电机关闭时清零**

```c
if (!motor_enable) {
    last_error = 0;
    pwm_out = 0;
    return 0;
}
```

这是一个踩过的坑（Session 3 修复）。如果电机关闭期间积分不清零，重新开启时残留的积分项会让车猛冲。

### 8. 平衡控制（control.c）— ★ 项目的核心

**技术点**：串级 PID、PD 控制器、安全保护

**设计思路**：
```
Balance_Control()  // 每 5ms 调用一次
  │
  ├─ 安全关停检查
  │   if (|pitch| > 30°) → turn_Off()  // 倒地了，别转了
  │
  ├─ 直立环 PD
  │   balance_pwm = -(Kp×(pitch-MECH_ZERO) + Kd×gyro_y)
  │   // pitch=倾角 → P项（弹簧），gyro_y=角速度 → D项（阻尼）
  │
  ├─ 速度阻尼（非积分）
  │   speed_sum = 左编码器 + 右编码器
  │   speed_pwm = SPEED_DAMP_K × speed_sum
  │   // 正向速度快 → 产生反向力，但死区内不响应
  │
  ├─ 叠加
  │   final = balance_pwm + speed_pwm
  │
  └─ 驱动电机
      motorMove(final, final)  // 两轮同向（平衡车不需要差速转弯）
```

**关键技术：为什么 balance_pwm 前面有负号？**

```c
balance_pwm = -(int)(Kp×pitch + Kd×gyro_y);
```

因为控制逻辑是"向倾倒方向加速"：
- 车往前倒（pitch > 0）→ 需要轮子往前转 → 产生反向力矩把车推回来
- 但根据电机接线和轴配置，正 PWM 可能对应的是往后转
- 负号在 Session 3 换轴后添加，确保电机转向正确

**关键技术：速度阻尼为什么不用积分？**

正式的速度环 PI 中有积分项 I，它会不断累积速度误差。但在直立环还没调好的时候，积分项会和直立环"打架"——车为了维持站立需要运动，但速度环的积分却说"你动了，回来！"，结果把平衡点推偏。

所以当前方案用**纯比例速度阻尼**（无积分）：速度快了就拉一下，但不强迫回到原位。等直立环稳定后再启用正式的速度环 PI。

**关键技术：调试变量**

```c
debug_balance_pwm, debug_speed_pwm,
debug_final_left, debug_final_right
```

这些变量记录控制计算的中间结果，由 main.c 的主循环通过串口输出。在调参阶段，观察这些值能判断是直立环还是限幅出了问题。

### 9. 主程序（main.c）— 初始化顺序 + 主循环 + 桩函数

**技术点**：外设初始化顺序、非阻塞循环、printf 重定向

**设计思路**：

**初始化顺序**（顺序不能乱）：
```c
HAL_Init()              ← ① 必须先初始化 HAL
sys_stm32_clock_init()  ← ② 配置系统时钟（64MHz）
delay_init()            ← ③ 延时功能依赖时钟，必须紧随其后
oled_init()             ← ④ OLED 优先初始化（后续 init 的状态能在屏上看到）
usart_init()            ← ⑤ 串口初始化（方便 printf 调试）
                          // 以上是系统基础设施
led_init()              ← ⑥ 其他外设初始化（顺序无所谓）
motor_init()
encoder_init()
gtim_init()
adc_init()
MPU6050_init()
exti_init()             ← ⑦ 中断最后初始化（确保所有依赖已就绪）
```

**主循环**（50ms 周期，非阻塞）：
```c
while (1) {
    if (HAL_GetTick() - last_show >= 50) {  // 非阻塞定时
        OLED 显示 4 行数据
        oled_refresh_gram()                 // 推送到屏幕
        printf CSV 调试数据                 // 串口输出
    }
}
```

为什么主循环只做显示，不做控制？因为控制需要严格的 5ms 周期，放在中断里更精确。主循环的 50ms 是显示刷新率（20Hz），人眼看起来足够流畅。

**关键技术：桩函数（Stub）**

```c
// 这些函数是 HAL 库的链接依赖，但我们不需要实际功能
HAL_StatusTypeDef HAL_DMA_Abort_IT(...)  { return HAL_OK; }  // 空函数体
uint32_t HAL_RCCEx_GetPeriphCLKFreq(...) { return 36000000; } // 返回假值
```

为什么需要？编译时链接器要求所有被引用的符号都存在。HAL_TIM 会引用 DMA 相关函数和 RCCEx 函数，但我们不用 DMA（省代码空间）。与其链接完整的 `hal_dma.c`（1.7KB），不如写空函数（几行代码）替代。

### 10. 死区测试程序（test2/main.c）— 独立测试工具

**设计思路**：在调参过程中发现死区问题后，写了一个独立的 Keil 工程，专门做 PWM 精细扫描（280-320 范围，步进 5），帮助找到每个电机的确切死区值。

## 开发过程中的关键踩坑与教训

| # | 问题 | 原因 | 教训 |
|---|------|------|------|
| 1 | 串口乱码 | HSE 晶振频率不是 8MHz，波特率偏了 | 不要假设硬件参数，要用内部可靠的时钟源 |
| 2 | OLED 不显示 | 只写了显存没调用 refresh | 显示芯片不会自动刷新，需要手动推送 |
| 3 | 右编码器无数据 | TIM3 CH1(PA7) 和 OLED CS(PA7) 冲突 | 分配引脚前要检查冲突，特别是 JTAG 引脚 |
| 4 | MPU6050 无中断 | INT 使能寄存器没开（0x00） | 参考实现不一定全对，要对照数据手册验证 |
| 5 | 电机方向反了 | 换轴后 balance_pwm 符号不对 | 换传感器配置后要重新验证整个控制链的极性 |
| 6 | 速度环推偏平衡点 | 在直立环不稳定时开了积分 | 串级 PID 必须"先内后外"，不可并行调 |
| 7 | PB3/PB4/PA15 HAL_GPIO_Init 失败 | 这些是 JTAG 引脚，HAL 库拒绝配置 | 特殊引脚需要用寄存器操作，且先 remap JTAG |
| 8 | 烧录后没反应 | HAL 状态机残留 | 烧录后给芯片断电重启 |
