## 1. 项目概述与目标

### 1.1 项目背景

二轮平衡小车是一种基于倒立摆原理的自平衡机器人系统。其核心挑战在于：车身本质上是**不稳定系统**（重心在轮轴之上），能够通过实时检测倾斜角度并通过电机驱动车轮，产生与倾倒方向相反的回正力矩，使车身维持动态平衡。

本项目基于 STM32F103C8T6 微控制器，使用 MPU6050 六轴姿态传感器（三轴加速度计 + 三轴陀螺仪）实时检测车身倾角，采用**串级 PID 控制算法**（内环直立 PD + 外环速度 PI），驱动 TB6612 电机驱动模块控制两个直流减速电机，实现小车的自平衡站立。

### 1.2 项目目标

**基本目标**（课堂要求）：
1. 小车能够**原地直立站立**，稳定时轮向位移 < 5cm
2. 具备**抗干扰能力**，受到轻推后能自动恢复直立
3. 电池电压低于 10V 时，平衡功能自动关闭
4. 按键控制电机启停
5. 在 OLED 上实时显示倾角、电池电压、电机状态等信息

**已完成情况**：
- ✅ 小车可在桌面上持久平衡，维持较小运动范围内稳定
- ✅ 所有驱动模块正常（MPU6050、电机 PWM、编码器、OLED、串口）
- ✅ 串级 PID 控制完整实现（内环 PD + 外环 PI）
- ✅ 串口实时输出 CSV 调试数据（200Hz 控制频率）
- ✅ OLED 实时状态显示
- ✅ 安全关停（倾角超 30° 自动关闭电机）
- ✅ 死区补偿与电机增益校准

**待完善项**：
- ⚠️ 左编码器硬件故障（始终为 0），影响速度环精度
- ⚠️ 存在轻微高频抖动，可进一步微调参数
- ⚠️ ADC 分压比需万用表校准后恢复低压保护

## 2. 项目方案

### 2.1 系统总体方案

系统采用**经典倒立摆控制架构**：传感器感知姿态 → 控制器计算回正力 → 执行器驱动电机。在 STM32 内部形成 **200Hz 的高速闭环控制循环**。

```
┌─────────────────────────────────────────────────────────┐
│                    二轮平衡小车系统                      │
│                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐           │
│  │ MPU6050  │───▶│ STM32F103│───▶│ TB6612   │──▶ 电机   │
│  │ 姿态传感器│    │ 主控制器  │    │ 电机驱动  │           │
│  └──────────┘    └────┬─────┘    └──────────┘           │
│                       │                                 │
│            ┌──────────┼──────────┐                      │
│            ▼          ▼          ▼                      │
│       ┌────────┐ ┌────────┐ ┌────────┐                  │
│       │ 编码器  │ │  OLED  │ │ 串口   │                  │
│       │(速度反馈)│ │(状态显示)│ │(调试输出)│                  │
│       └────────┘ └────────┘ └────────┘                  │
└─────────────────────────────────────────────────────────┘
```

### 2.2 控制方案选择

采用**串级 PID 控制**，分内外两环：

- **内环——直立环 PD**：根据倾角 (pitch) 和角速度 (gyro_y) 计算回正力矩。P 项提供角度比例恢复力，D 项提供角速度阻尼（"刹车"效果），抑制振荡。
- **外环——速度环 PI**：根据编码器测得的速度反馈，消除稳态速度漂移，使车保持在原地。

控制公式：
```
balance_pwm = -(Kp × (pitch - MZ) + Kd × gyro_y)     (直立环 PD)
speed_pwm   = Vel_PI(speed_left + speed_right, 0)    (速度环 PI)
final_pwm   = PWM_Limit(balance_pwm + speed_pwm)     (叠加 + 限幅)
```

#### 3.2.2 五大核心文件的协作关系

系统平衡功能的实现由**5 个核心文件**协同完成。它们通过**硬件中断链 + 全局变量共享 + 函数调用**三种机制紧密配合：

| 文件 | 角色 | 职责 |
|------|------|------|
| `main.c` | **指挥官** | 初始化所有外设（严格顺序），然后进入主循环做 OLED 显示和串口输出 |
| `exti.c` | **心跳起搏器** | 响应 MPU6050 的 5ms INT 中断，读取传感器数据、互补滤波、触发控制循环 |
| `control.c` | **大脑** | 执行串级 PID 控制算法，计算电机目标 PWM |
| `motor.c` | **肌肉** | 接收 PWM 指令，做死区补偿和增益校准，输出实际 PWM 到 TIM1 |
| `gtim.c` | **脉搏计数器** | TIM3 5ms 中断读取编码器速度，提供增量式 PI 速度环函数 |

**全局变量数据流**（跨文件通信方式）：

```
生产者               全局变量                消费者
──────────────────────────────────────────────────
exti.c  ──→  pitch, gyro_y          ──→  control.c, main.c
exti.c  ──→  battery_voltage        ──→  main.c
exti.c  ──→  balance_enable         ──→  control.c
exti.c  ──→  motor_enable           ──→  motor.c, control.c, gtim.c
gtim.c  ──→  speed_left, speed_right ──→  control.c, main.c
control.c ──→ debug_balance_pwm 等   ──→  main.c (串口输出)
motor.c   ──→ motor_left, motor_right ──→  main.c
```

> 使用全局变量的原因：数据生产者是中断服务函数（ISR），中断由硬件触发，没有"调用者"可以传递参数。ISR 修改全局变量、其他代码读取——这是嵌入式系统中最常见的 ISR 通信模式。

#### 3.2.4 控制循环时序图（200Hz 实时闭环）

系统中有**两条并行的 200Hz 时间线**，由不同硬件触发、彼此独立：

```
时间线 A（传感器 → 控制 → 电机）：
  MPU6050 内部 200Hz 时钟
    → INT 引脚 (PA12) 拉低（每 5ms）
    → EXTI15_10 中断 [优先级 2]
    → exti.c: 读 MPU6050 → 互补滤波 → pitch, gyro_y
    → control.c: Balance_Control() → PD + PI → PWM
    → motor.c: motorMove() → 死区补偿 → TIM1 CCR

时间线 B（编码器读取）：
  STM32 TIM3 计数器溢出（每 5ms）
    → TIM3 中断 [优先级 1]（可抢占时间线 A）
    → gtim.c: encoder_read_all() → speed_left, speed_right
```

> TIM3 优先级更高（1 > 2），确保编码器数据始终最新。

## 4. 关键技术及程序设计说明

### 4.1 姿态解算：互补滤波

#### 4.1.1 原理

MPU6050 提供两类传感器数据：
- **加速度计**：通过重力分量 `atan2f(ay, az)` 计算倾角。静态准确但动态时受运动加速度干扰，噪声大。
- **陀螺仪**：对角速度积分得到角度变化量。短期精确但存在零漂，长时间积分会发散。

**互补滤波器**本质是一个**加权平均**——高通滤陀螺仪（信任短期变化），低通滤加速度计（信任长期参考）：

```
comp_angle = α × (comp_angle + gyro × Δt) + (1 - α) × accel_angle
```

其中 `α = 0.95`，`Δt = 0.005s`。陀螺仪积分占 95% 权重（平滑、响应快），加速度计占 5%（缓慢修正漂移）。

#### 4.1.2 代码实现（[exti.c:63-74](Template/Drivers/BSP/EXTI/exti.c#L63-L74)）

```c
// 读取 MPU6050 原始数据（通过软件 I2C）
int16_t ax, ay, az, gx, gy, gz;
static float comp_angle = 0;  // static! 跨中断调用保持累积值

atk_ms6050_get_accelerometer(&ax, &ay, &az);
atk_ms6050_get_gyroscope(&gx, &gy, &gz);

// 加速度计 → 倾角（atan2f 返回弧度，×57.3 转度数）
float accel_angle = atan2f((float)ay, (float)az) * 57.29578f;

// 陀螺仪 → 角速度（÷16.4：±2000°/s 量程，16.4 LSB/°/s）
gyro_y = (float)gx / 16.4f;

// 互补滤波
comp_angle = 0.95f * (comp_angle + gyro_y * 0.005f) + 0.05f * accel_angle;
pitch = comp_angle;
```

**关键设计点**：
- `comp_angle` 为 `static` 局部变量：生命周期跨越每次中断调用，维持上一次融合结果供陀螺仪积分。
- 互补滤波放在 `exti.c` 而非 MPU6050 驱动中：**传感器驱动职责是读数据**，姿态解算属于应用层逻辑。更换滤波算法时无需修改驱动。
- DLPF 默认 98Hz 带宽，对电机振动有一定抑制。

### 4.2 串级 PID 控制算法

#### 4.2.1 为什么用串级 PID？

单级 PID（一个角度环直接控制 PWM）理论上也可以工作，但存在以下问题：
- 无法区分"因倾角回正"和"因速度漂移"两种场景
- 小车可能站立但缓慢朝一个方向移动（速度静差）
- 同时处理"角度"和"速度"两种不同时间尺度的误差不理想

**串级结构**：内环（直立 PD）高频响应角度变化（200Hz），外环（速度 PI）低频修正位置漂移。两环职责分离，各自参数独立调节。

#### 4.2.2 直立环 PD

**P（比例项）**：根据角度偏差产生与偏差成正比的回正力。倾角越大，回正力矩越大。

**D（微分项）**：根据角速度（角度变化率）产生阻尼力——**在倾倒刚开始时就提前响应**，而不是等到倾角很大才动作。本质是"预见性刹车"。

```c
// control.c:55
balance_pwm = -(int)(Balance_Kp * (pitch - MECHANICAL_ZERO) + Balance_Kd * gyro_y);
```

例：车以 50°/s 前倾且倾角为 2°时：
- P 项 = 380 × (2.0 - (-0.9)) = 380 × 2.9 = 1102
- D 项 = 23.5 × 50 = 1175
- 合计 = -(1102 + 1175) = -2277（反向力）

**D 项的作用远超 P 项**——在倾角还没有大到危险时，角速度已经很大，D 项立即输出强大的刹车力。这是平衡控制的核心。

#### 4.2.3 速度环 PI —— 增量式

增量式 PI 公式：
```
ΔU = Kp × (e(k) - e(k-1)) + Ki × e(k)
U(k) = U(k-1) + ΔU
```

**代码实现（[gtim.c:33-64](Template/HARDWARE/TIMER/gtim.c#L33-L64)）**：

```c
int Vel_PI(int vel, int Target)
{
    int error = Target - vel;
    static int last_error = 0;     // 上一次误差
    static int pwm_out = 0;        // 累计输出值

    // 电机关闭时清零，防止重启后猛冲（积分清零 + 历史清零）
    if (!motor_enable) {
        last_error = 0;
        pwm_out = 0;
        return 0;
    }

    int delta = (int)(Velocity_Kp * (error - last_error) + Velocity_Ki * error);
    pwm_out += delta;

    // 抗积分饱和：限幅到 PWM_MIN ~ PWM_MAX
    if (pwm_out > PWM_MAX) pwm_out = PWM_MAX;
    else if (pwm_out < PWM_MIN) pwm_out = PWM_MIN;

    last_error = error;
    return pwm_out;
}
```

**增量式 PI 的优势**：
- **自动抗积分饱和**：输出限幅后，`pwm_out` 不会无限制增长
- **无冲击切换**：电机关闭时清空 `last_error` 和 `pwm_out`，重新启动不会突然猛冲
- **天然安全**：`static` 变量保持状态，即使积分累计也不担心

#### 4.2.4 最终参数

经多轮调参确定的最终值（[control.c](Template/HARDWARE/CONTROL/control.c)）：

| 参数 | 值 | 说明 |
|------|-----|------|
| `Balance_Kp` | 380.0 | 直立环比例——提供角度恢复力 |
| `Balance_Kd` | 23.50 | 直立环微分——角速度阻尼，Kd/Kp=0.062 |
| `velocity_Kp` | 140 | 速度环比例——速度漂移修正 |
| `Velocity_Ki` | 0.70 | 速度环积分——消除静差 |
| `MECHANICAL_ZERO` | -0.90° | 物理平衡点偏移（重心偏后） |
| `SAFE_ANGLE_MAX` | 30° | 超过此倾角自动关停电机 |

### 4.3 电机死区补偿

#### 4.3.1 问题背景

直流电机存在**启动死区**——PWM 占空比低于一定值时，电机无法克服静摩擦力而无法转动。通过 `test2/` 测试程序实测，本车左右电机死区约 280~285（占 7200 的约 4%）。

直接补偿死区有**三种方案**，在调试中逐一尝试：

| 方案 | 逻辑 | 问题 |
|------|------|------|
| 硬阈值 | `if (pwm>0 && pwm<DZ) pwm=DZ;` | 小角度下"要么不转要么过猛"，无中间地带 |
| 偏移式 | `if (pwm>0) pwm+=DZ;` | 微小噪声也被放大到 DZ 级别，"手都按不住" |
| **柔和死区**（当前） | `if (pwm>5) pwm+=100; else if (pwm<-5) pwm-=100;` | 小信号不响应（靠自然摩擦），中等以上信号加 100 |

**当前柔和死区实现（[motor.c:31-39](Template/Drivers/BSP/MOTOR/motor.c#L31-L39)）**：

```c
static void motor_deadzone(int *pwm, int dz)
{
    if (*pwm > MOTOR_DZ_START)       // pwm > 5
        *pwm += dz;                   // +100
    else if (*pwm < -MOTOR_DZ_START)  // pwm < -5
        *pwm -= dz;                   // -100
    else
        *pwm = 0;                     // |pwm| ≤ 5 → 不转
}
```

**设计思路**：
- `MOTOR_DZ_START = 5`：信号太小（噪声级别）不驱动，靠机械摩擦保持静止
- `MOTOR_DZ = 100`：一旦决定要转，额外补偿 100 确保越过死区
- 为什么不补偿满 280？PID 的 P 项本身也在输出，叠加后足够克服静摩擦，100 刚好弥补"电机从不动到能动的最大缺口"

### 4.4 OLED 寄存器级驱动

#### 4.4.1 问题

PB3/PB4/PA15 是 JTAG 调试引脚，HAL 的 `HAL_GPIO_Init()` 函数在配置 JTAG 引脚时会因 AFIO 锁定而失败。且项目代码量接近 MDK 评估版 32KB 限制（代码空间仅剩 636 字节），需要极致精简。

#### 4.4.2 解决方案

弃用 HAL GPIO API，改为**直接操作 GPIO 寄存器 BSRR/BRR**控制 OLED 引脚电平，绕过了 HAL 初始化限制。同时弃用 HAL_ADC 改为寄存器版 ADC，DMA 用空桩替代，移除不使用的 HAL 模块（tim_ex、adc_ex、rcc_ex、pwr、spi 等），将代码从 55KB 压缩到 32KB。

### 4.5 代码空间优化

| 优化措施 | 节省空间 |
|----------|----------|
| MicroLIB + -Os 优化 | ~5KB |
| 移除 OLED 12×12 和 24×24 字体 | ~450B |
| HAL_ADC → 寄存器版重写 | ~740B |
| HAL_DMA → 空桩函数 | ~1.7KB |
| 移除 HAL 扩展模块 | ~2KB |
| **最终代码大小** | **30300B + 1832B(RO) = 32132B** |

### 5.2 关键 Bug 及修复记录

以下是调试过程中遇到的最关键问题和修复方案：

#### Bug 1: 右编码器与 OLED 引脚冲突

**现象**：右编码器读数为 0，OLED 显示异常。

**原因**：右编码器最初配置在 TIM3 (PA6/PA7)，而 OLED CS 也用 PA7。两者共用同一 GPIO 引脚，发生冲突。

**修复**：将右编码器从 TIM3 迁移到 TIM4 (PB6/PB7)，原 TIM3 改为控制定时器使用。修改涉及 `encoder.c/h`、`gtim.c` 两个文件，重新分配所有定时器角色。

#### Bug 2: MPU6050 INT 中断未触发

**现象**：控制循环不执行，电机无响应。

**原因**：MPU6050 初始化代码中 INT 使能寄存器写入 0x00（未使能中断输出），且默认采样率为 50Hz 而非 200Hz。

**修复**（`atk_ms6050.c`）：
- 采样率配置改为 200Hz：`atk_ms6050_set_rate(200)`
- INT 使能寄存器：`0x00 → 0x01`

#### Bug 3: HSE 晶振不可用导致死锁

**现象**：烧录后程序在 `while(1)` 处死循环，芯片完全无响应。

**原因**：`sys.c` 中系统始终初始化函数等待外部 HSE 晶振起振，如果晶振不存在或损坏→`while(1)` 永不出。

**修复**：添加 HSE 失败自动回退逻辑——HSE 起振超时后改用内部 HSI+PLL 产生 64MHz 时钟：
```c
if (HAL_RCC_OscConfig(&RCC_OscInitStruct) != HAL_OK) {
    // 回退到 HSI → HSI/2 × 16 = 64MHz
    RCC_OscInitStruct.OscillatorType = RCC_OSCILLATORTYPE_HSI;
    ...
}
```

#### Bug 4: OLED 不显示——缺少 refresh 调用

**现象**：OLED 初始化成功但不显示字符。

**原因**：oled.c 的 `oled_show_string()` 只将数据写入内部显存缓冲区，但主循环遗漏了 `oled_refresh_gram()` 调用。

**修复**：在 main.c 主循环中添加 `oled_refresh_gram()`，将缓冲区数据实际推送到 SSD1306 硬件。

#### Bug 5: MPU6050 轴配置错误

**现象**：小车前倾时 pitch 符号与预期相反或完全不对。

**原因**：MPU6050 安装方向与标准不同，使用的轴组合（ay + gx）与代码初始假设不同。

**修复**：通过多次实验确认采用 AY + GX 轴组合，并将 balance_pwm 取反（加负号）以匹配电机接线方向。

#### Bug 6: PWM 限幅卡死控制力

**现象**：串口数据显示控制器输出的 `balance_pwm` 被限制在 ±800，但小车仍缓慢倒下。

**原因**：`MOTOR_SAFE_LIMIT = 800`（调参初期为防止猛冲设置的保护值）在 Kp 增大后成为了"天花板"——控制器想输出 1500 但被裁剪到 800。

**修复**：将 `MOTOR_SAFE_LIMIT` 逐步提高到 7199（硬件最大占空比），解除控制器的"手脚束缚"。

### 5.3 PID 调参过程

#### 5.3.1 调参总原则

```
先内环（直立 PD），后外环（速度 PI）
每次只改一个参数，观察效果后再改下一个
Kd/Kp 比值是最关键参数
```

#### 5.3.2 调参里程碑

| 阶段 | Kp | Kd | Kd/Kp | 结果 |
|------|-----|-----|-------|------|
| Session 4 初始 | 250 | 0.5 | 0.002 | 无力，缓慢倒下 |
| Session 5 | 260 | 2.0 | 0.008 | PWM 顶到限幅 800，仍倒 |
| Session 6 探索 | 240~350 | 1.35~7.0 | 0.005~0.025 | 发现 Kd/Kp<0.018 不收敛 |
| Session 6 最佳 | 300 | 7.0 | 0.023 | 振荡 ±1.6°，4 秒后增长倒下 |
| Session 7 前期 | 280~350 | 5.5~6.0 | 0.017~0.02 | 站立 6~7 秒后振幅缓慢增长 |
| **Session 7 最终** | **380** | **23.5** | **0.062** | **持久平衡，小幅高频抖动** |

#### 5.3.3 关键发现

1. **Kd/Kp 比值是本车最关键参数**。理论书籍通常给出 Kd/Kp = 0.002~0.01，但从其他组获取的参考参数（Kd/Kp=0.0053）在本车上无法收敛。经 6 轮迭代，最终发现本车需要 **Kd/Kp ≈ 0.062** 才能持久平衡。这与车的重心位置、电机功率、轮径等机械参数密切相关，不能直接套用通用公式。

2. **速度阻尼 ≠ 速度环**。Session 6 中使用纯比例速度阻尼（`SPEED_DAMP_K × speed_sum`），只能抑制快速跑远，无法消除慢速漂移。Session 7 接入增量式 PI 后，速度漂移被有效抑制。

3. **PWM 限幅解除是突破性的一步**。早期 `MOTOR_SAFE_LIMIT=800` 限制了控制器的最大出力，Kp 即使调大也无法体现效果。

4. **死区补偿不能照搬理论值**。实测死区 285，但直接偏移 285 会产生剧烈抖动。柔和补偿（start=5, dz=100）在响应性和稳定性间取得了平衡。

## 6. 项目总结

### 6.1 项目过程回顾

本项目从零开始构建了一个完整的二轮平衡小车系统。在 7 个 session 中完成了**全部 32 个源文件的编写与调试**，覆盖了从 CMSIS 启动到 PID 控制的所有层级。

核心技术链：**CMSIS → HAL 库 → BSP 外设驱动 → SYSTEM 系统模块 → HARDWARE 控制算法 → User 主程序**，体现了嵌入式系统的经典分层架构。

### 6.3 主要收获

1. **倒立摆控制的工程实践**：从串级 PID 理论到实际调参，深刻理解了 Kd/Kp 比值的重要性以及"纸上理论值"与"实物机械参数"之间的差距。

2. **嵌入式系统分层设计**：学会了将系统拆分为 CMSIS 层、HAL 库层、BSP 驱动层、HARDWARE 算法层、User 应用层，每层职责分明，修改影响范围可控。

3. **中断驱动架构**：掌握了通过硬件中断链（MPU6050 INT → EXTI → exti.c → control.c → motor.c）构建实时闭环控制的完整方法，理解了 NVIC 优先级在实时系统中的关键作用。

4. **互补滤波 vs 复杂滤波**：简单的互补滤波（仅 1 行代码）在实践中就足够稳定，不需要卡尔曼滤波等复杂算法——工程实践中应选择与需求匹配的算法复杂度。

5. **死区补偿的经验价值**：课本上只是提到"需要补偿"，但补偿多少、用什么方式（偏移/硬阈值/柔和），完全依赖实验确定。这是纯工程经验的积累。

6. **代码空间优化**：在 MDK 评估版 32KB 限制下，学会了 MicroLIB、-Os 优化、寄存器驱动替代 HAL、字体裁剪、空桩函数等多种压缩技术。

### 6.4 展望与改进

1. **左编码器修复**：当前 `speed_left` 始终为 0（PA0/PA1 或 TIM2 硬件问题），修复后速度环精度将显著提升。

2. **转向控制**：当前左右轮 PWM 完全相同，无转向差速。可增加陀螺仪 Z 轴偏航控制，实现方向锁定或遥控转向。
